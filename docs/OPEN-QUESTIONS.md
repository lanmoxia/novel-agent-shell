# 不确定项记录

> 本文档记录开发过程中遇到的不确定项。每个条目包含问题、当前保守方案和后续建议。

---

## 未决问题

### Q1：Claude Code 运行配置是否还需要保留
- **问题**：项目最初由 Claude Code 创建，`.claude/settings*.json` 属于运行配置，不是流程规则资产。
- **保守方案**：已清除 `.claude/` 运行目录，并将流程资产迁移到 `workflow/`。
- **后续建议**：后续新增流程规则统一放入 `workflow/`，不要恢复 `.claude/`。

### Q2：题材 Skill 文件的"按需读取"粒度
- **问题**：规格要求"按需读取"，但未明确每次读取多少。例如写对白时需要读 dialogue-rules.md，但是否需要同时读 character-rules.md？
- **保守方案**：当前阶段定义 genre-loader 读取所有可用文件并生成摘要到 GENRE-SKILL-LOCK.md。后续阶段优先读取摘要文件。具体哪个阶段读哪些文件，在各 Skill 的指令中显式声明。
- **后续建议**：在 Phase 2 实际的 genre-loader 中细化。

### Q3：feedback-observer 的触发机制
- **问题**：反馈检测通过 Hook（detect-feedback-signal.sh）触发，但 Hook 只能通过关键词判断。实际反馈识别需要语义理解。
- **保守方案**：Hook 只提供提示（"检测到用户反馈信号"），实际判断由 feedback-observer Agent 完成。Hook 不自动触发 Agent，由用户或主流程决定。
- **后续建议**：观察实际使用中反馈识别的准确率。

### Q4：多项目并行创作的支持
- **问题**：当前规格假设一次只创作一个项目。如果用户同时维护多个项目（如一部小说 + 一部剧本），当前结构是否足够？
- **保守方案**：`projects/` 目录天然支持多项目（每个项目一个子目录）。同一维护会话只操作一个项目。
- **后续建议**：如果多项目需求明确，需要增加项目切换说明。

### Q5：题材 Skill 的版本管理
- **问题**：题材 Skill 可能被更新，但已锁定的项目应该继续使用旧版本还是跟随更新？
- **保守方案**：GENRE-SKILL-LOCK.md 锁定的是加载时间的具体版本（记录已读取文件的摘要）。已锁定项目不跟随题材 Skill 更新。如需使用新版，必须新建项目。
- **后续建议**：如果需要题材 Skill 版本化，可以给题材 Skill 加 version 字段。

### Q6：Token 消耗的实际测量
- **问题**：规格中的 TOKEN-CONTROL.md 规则是经验性的，没有经过实测。
- **保守方案**：保持当前规则。在 Phase 2 完成后用一个模拟项目测试 token 消耗。
- **后续建议**：在反馈系统中增加 token 消耗监控。

- **结论**：保留 Bash hook 和 Bash 验证入口，同时新增 PowerShell 维护入口。
